# Horizon 三级侧边触发器设计

## 目标

解决当前缩小状态占用面积过大、无法可靠上下拖动的问题。缩小后的 Horizon 使用三级形态：极细触发条、悬停把手、完整面板。

## 范围

本次只调整右侧面板的显示状态、鼠标交互、纵向拖动和状态切换动画。任务数据、编辑器、项目归档和完整面板内部布局保持不变。

## 实现方式

继续使用一个 WPF `Window`，由显式状态控制窗口边界和可见外壳。相比透明大窗口或双窗口方案，单窗口不会遮挡额外桌面区域，也避免焦点、置顶层级和多屏生命周期变复杂。

状态定义：

- `CollapsedSliver`：常驻极细触发条。
- `HoverHandle`：鼠标悬停后出现的可操作把手。
- `ExpandedPanel`：完整任务面板。

任一时刻只能存在一个状态。

## 尺寸与布局

- 极细触发条：`6 × 72 px`，贴紧工作区最右边。
- 悬停把手：`30 × 92 px`，从右边缘向左展开。
- 完整面板：保持现有 `360 px` 宽，覆盖当前工作区高度。
- 极细触发条和悬停把手共用同一纵向中心位置。
- 触发器纵向移动范围距离工作区顶部和底部至少 `20 px`。

## 交互

1. 程序收起时显示极细触发条。
2. 鼠标进入极细触发条后，用 `120 ms` 展开为悬停把手。
3. 鼠标离开悬停把手后等待 `250 ms`；若鼠标没有返回，则缩回极细触发条。
4. 单击悬停把手，展开完整面板。
5. 在悬停把手上按住鼠标并纵向移动超过 `4 px` 后进入拖动；拖动期间窗口持续跟随鼠标。
6. 松开鼠标后结束拖动并保存纵向位置。发生过拖动的手势不得同时触发展开。
7. 完整面板点击“收起”后，返回之前保存位置上的极细触发条。

极细触发条只负责触发悬停，不直接承担拖动或单击操作，避免过小命中区域导致误操作。

## 状态与异常处理

- 拖动期间暂停离开缩回计时器，并保持鼠标捕获；即使指针移出把手也继续拖动。
- 鼠标捕获意外丢失时立即结束拖动，保留当前位置，但不触发展开。
- 工作区尺寸或任务栏位置变化时，重新限制已保存位置，确保触发器仍在可见范围内。
- 动画进行中以目标状态为准，新的状态切换会覆盖旧动画，避免可见外壳和窗口尺寸不一致。
- 启动时读取已有的 `CollapsedButtonTop`；若超出当前工作区则先修正再显示。

## 数据与兼容性

继续复用现有 `CollapsedButtonTop` 设置，不改变本地 JSON 数据结构。该值明确表示悬停把手的顶部坐标；极细触发条的顶部坐标在此基础上增加 `10 px`，从而让两种形态保持同一纵向中心。旧版本保存的位置直接作为新悬停把手的顶部坐标使用，并在首次显示时按新高度限制到工作区内。

## 验证

- 极细触发条尺寸和右侧贴边位置正确。
- 悬停能展开把手，离开延迟后能缩回，快速移出再返回不会闪烁。
- 把手单击展开完整面板。
- 把手可上下拖动，拖动不会误触发展开。
- 拖动到顶部和底部时遵守 `20 px` 边距。
- 鼠标捕获丢失后不会卡在拖动状态或误展开。
- 收起面板后返回原纵向位置，重启程序后位置仍保留。
- 分辨率或工作区变化后触发器保持可见。
- `dotnet build Horizon.sln` 通过。
- 发布到 `artifacts\Horizon.App` 后启动 EXE，手工验证完整交互闭环。

## 非目标

- 不修改完整面板内的任务列表、状态控件或编辑流程。
- 不增加吸附到左侧、任意桌面位置或多触发器支持。
- 不增加新的用户设置项。
