# Horizon 点击外部自动收起设计

## 目标

移除完整面板顶部的“收起”按钮。用户点击 Horizon 主界面以外的桌面、任务栏或其他应用时，完整面板自动收起为极细触发条。

## 范围

本次只调整完整面板的关闭入口和窗口失活行为。三级触发器尺寸、悬停展开、纵向拖动、任务数据和编辑表单保持不变。

## 交互规则

- 完整面板不再显示“收起”按钮。
- 完整面板处于 `ExpandedPanel` 状态时，窗口失去激活即自动切换到 `CollapsedSliver`。
- 点击桌面、任务栏或其他应用会触发自动收起。
- 使用 Alt+Tab 切换到其他应用也会触发自动收起。
- 应用自身的上下文菜单或弹出控件不视为外部操作，不应导致面板收起。
- `Esc` 行为保持不变：编辑器打开时先取消编辑；没有编辑器时可收起完整面板。

## 编辑与草稿

如果用户正在新建或编辑任务，点击外部仍然收起完整面板，但不调用取消编辑，也不清空表单。再次从右侧触发器打开面板时，继续显示原来的编辑器和未保存草稿。

自动收起只改变面板显示状态，不写入、删除或重置任务数据。

## 实现方式

- 在 `MainWindow` 上监听 WPF 原生 `Deactivated` 事件。
- 事件发生后使用 `Dispatcher.BeginInvoke` 和 `DispatcherPriority.ApplicationIdle` 调度一次检查，让应用自身的菜单或弹出控件先完成状态更新。
- 检查时如果窗口已经重新激活、当前不再是 `ExpandedPanel`，或“新增内容”的 `ContextMenu` 仍然打开，则不收起；否则直接切换到 `CollapsedSliver`。
- 外部点击路径绕过现有的“编辑器打开时不允许普通收起”保护，但保留 ViewModel 中的编辑状态。
- 删除 XAML 中的“收起”按钮和不再使用的按钮事件处理方法。
- 不使用全局鼠标钩子、轮询或额外系统权限。

## 异常与边界

- 面板已经处于触发条或悬停把手状态时，窗口失活不重复切换状态。
- 快速重新激活窗口时，延迟检查以最终激活状态为准，避免闪烁。
- “新增内容”的 `ContextMenu` 是当前唯一可能影响窗口激活的应用内菜单，显式检查其 `IsOpen` 状态；普通 WPF 表单弹出控件不获取顶层窗口激活，不会触发此路径。
- 自动收起继续使用现有动画和原纵向触发器位置。

## 验证

- 完整面板顶部没有“收起”按钮，其他操作按钮布局正常。
- 点击桌面、任务栏和其他应用时，完整面板自动收起。
- Alt+Tab 切换应用时，完整面板自动收起。
- 打开“新增内容”菜单并选择菜单项时，面板不误收起。
- 编辑任务时点击外部会收起；重新打开后草稿和编辑器仍存在。
- `Esc` 仍先取消编辑，再收起面板。
- 已收起状态失活不会产生额外动画或状态错误。
- 自动测试、`dotnet build Horizon.sln` 和 Release 发布通过。

## 非目标

- 不增加新的关闭按钮、托盘菜单或设置项。
- 不改变任务保存、取消和校验规则。
- 不引入全局鼠标监听。
